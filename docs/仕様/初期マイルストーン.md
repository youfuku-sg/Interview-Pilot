# 初期マイルストーン

## 1. 目的

この文書は、Interview-Pilot を実際にアプリとして触れる状態にするまでの初期目標を整理する。

この段階では、面接支援機能の本格追加や Pluely 由来機能の削除には入らない。まずは、既存アプリを日本語で理解でき、インストール可能な成果物として使える状態にすることを目標とする。

## 2. 初期目標

まずは以下の 5 つを完了させる。

1. 主要 UI を日本語化する
2. README や資料を個人利用前提に整理する
3. GitHub Actions でインストーラを作れるようにする
4. GitHub Release からインストールして実際に触れるようにする
5. Pluely 由来の既存機能を把握する

ここまで完了したら、次の段階として不要機能の削除と面接支援機能の追加に進む。

## 3. この段階でやらないこと

- Pluely API / license 機能の削除
- analytics / telemetry の削除
- 面接セッション機能の追加
- 職務経歴書や求人票の登録機能追加
- 質問抽出や意図分類の実装
- 回答方針生成の実装
- ローカル STT / ローカル LLM の本格接続
- データベース構造の大きな変更

ただし、調査中に削除候補や追加候補を見つけた場合は、TODO やメモに追記する。

## 4. ステップ 1: 主要 UI を日本語化する

### 4.1 目的

Pluely の既存機能を実際に触って把握しやすくするため、主要 UI を日本語化する。

### 4.2 対象

- サイドバーメニュー
- メインオーバーレイ
- ダッシュボード
- チャット履歴
- システムプロンプト
- アプリ設定
- 回答設定
- スクリーンショット設定
- 音声設定
- カーソル / ショートカット設定
- エラー文言
- 空状態メッセージ
- 確認ダイアログ

### 4.3 方針

- 最初は i18n 化せず、直書き文言の日本語化でよい
- 機能名は必要に応じて英語を残す
- stealth / undetectable 系の文言は「目立たない表示」「非表示」「表示設定」などに置き換える
- Pluely API や license 関連の UI は、いったん意味が分かるように訳す
- 後続で削除するかどうかは、実際に触ってから判断する

### 4.4 完了条件

- 主要画面を日本語で迷わず操作できる
- どの設定が何を意味するか分かる
- Pluely 由来の英語文言が残っていても、機能把握に支障がない

## 5. ステップ 2: README や資料を個人利用前提に整理する

### 5.1 目的

リポジトリを開いたときに、これは Pluely そのものではなく、個人利用の Interview-Pilot 改修プロジェクトであることが分かるようにする。

### 5.2 対象

- `README.md`
- `SECURITY.md`
- `.github` 配下のテンプレート類
- `docs/仕様` 配下の資料

### 5.3 方針

- Pluely のマーケティング文言を整理する
- 個人利用前提であることを書く
- 配布、SaaS 化、商用提供は現時点で想定していないことを書く
- ローカル STT / ローカル LLM を将来の基本方針として書く
- 現時点では Pluely 由来の機能が多く残っていることを書く
- docs 配下の資料へ辿れるようにする

### 5.4 完了条件

- README を見れば、このリポジトリの目的が分かる
- Pluely の紹介ページではなく、Interview-Pilot の作業入口になっている
- 自分が数週間後に見返しても、次に何をするか分かる

## 6. ステップ 3: GitHub Actions でインストーラを作れるようにする

### 6.1 目的

ソースから毎回手元でビルドしなくても、GitHub Actions でインストール可能な成果物を作れるようにする。

### 6.2 対象

- `.github/workflows/publish.yml`
- `src-tauri/tauri.conf.json`
- `package.json`
- `src-tauri/Cargo.toml`
- 必要に応じて `VERSION`
- 必要に応じて `CHANGELOG.md`

### 6.3 方針

- まずは Windows 向けインストーラを優先する
- macOS / Linux は必要に応じて対象に含める
- Pluely 由来の secret 依存を整理する
- 署名なしビルドで始める候補とする
- Tauri updater JSON は最初は必須にしない
- GitHub Actions は手動実行または tag push を候補とする
- Release は draft を基本候補とする

### 6.4 完了条件

- GitHub Actions が成功する
- Windows 用インストーラが生成される
- 生成された成果物が GitHub Release に添付される
- 手順が README または docs から確認できる

## 7. ステップ 4: Release からインストールして触る

### 7.1 目的

GitHub Release に添付された成果物から実際にインストールし、アプリとして触れる状態にする。

### 7.2 確認内容

- インストーラをダウンロードできる
- Windows にインストールできる
- アプリを起動できる
- メインオーバーレイが表示される
- ダッシュボードを開ける
- 主要画面に移動できる
- 設定が保存される
- 会話履歴が保存される

### 7.3 注意点

- 署名なしの場合、Windows SmartScreen の警告が出る可能性がある
- Pluely 由来のアプリ名やアイコンが残っている可能性がある
- ライセンスや Pluely API 関連の UI が残っている可能性がある

### 7.4 完了条件

- Release から取得した成果物でアプリを起動できる
- 日本語化した主要 UI を実機で確認できる
- 以降の機能把握をインストール済みアプリで進められる

## 8. ステップ 5: Pluely 由来の既存機能を把握する

### 8.1 目的

既存機能を実際に触り、面接支援アプリに残すもの、削るもの、転用できるものを判断する。

### 8.2 確認対象

- メインオーバーレイ
- AI チャット
- システムプロンプト
- 会話履歴
- 音声入力
- システム音声取得
- スクリーンショット
- ファイル添付
- 回答設定
- アプリ設定
- カーソル / ショートカット
- Pluely API / license
- analytics / telemetry

### 8.3 記録すること

各機能について、以下を記録する。

- 何ができるか
- 実際に動いたか
- 設定が必要か
- 面接支援に使えそうか
- 削除候補か
- 後で調査が必要か

### 8.4 完了条件

- 主要機能を一通り触っている
- 残す機能、削る候補、転用候補が見えている
- 次フェーズである「不要機能の整理」と「面接支援機能の追加」に進める

## 9. 初期マイルストーン完了後の次フェーズ

初期マイルストーン完了後は、以下に進む。

1. 不要機能の整理
2. Pluely API / license / analytics の扱い整理
3. ローカル LLM 接続
4. ローカル STT 接続
5. 面接セッション機能
6. 事前資料管理
7. 質問抽出と意図分類
8. 回答方針生成
9. 面接中オーバーレイ
10. 面接後振り返り
