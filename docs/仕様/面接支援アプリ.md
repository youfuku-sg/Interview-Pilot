# 面接支援アプリ 調査メモ

## 1. 調査目的

面接中の質問をリアルタイムで文字起こしし、質問意図を解析したうえで、事前に登録した職務経歴書・自己PR・応募企業情報などをもとに回答方針を提示するアプリが、実用レベルで成立するかを調査する。

## 2. 先に整理しておきたい前提

- これは仕様確定用のファイルではなく、調査メモとして使う
- 実現可能性、使い勝手、法的・倫理的な懸念、技術選定を順に詰める
- まずは広めに論点を出し、その後MVPに絞る

## 3. 作りたいものの仮説

- 応募者が面接中に質問の意図を見失わず、回答の方向性を短時間で掴めると価値がある
- 緊張で頭が真っ白になる場面では、長文生成よりも短い要点提示の方が有効そう
- 面接中支援だけでなく、面接前準備と面接後振り返りまで含めると継続利用されやすそう

## 4. 想定ユーザー

- 転職活動中のエンジニア
- 面接が苦手な応募者
- 職務経歴をうまく言語化したい人
- 面接前後の振り返りを効率化したい人

## 5. 想定ユースケース

### 5.1 面接前

- 職務経歴書、自己PR、志望動機、求人票、応募企業情報を登録する
- AIが資料を要約し、面接で使える論点を整理する
- 想定質問と回答メモを貯める

### 5.2 面接中

- 面接官の質問をリアルタイムで文字起こしする
- 質問意図を分類する
- 登録済み資料を参照し、回答方針や短い回答例を提示する

### 5.3 面接後

- 質問一覧や回答履歴を見返す
- うまく答えられなかった質問を抽出する
- 次回までの改善ポイントを整理する

## 6. 調査したい機能候補

### 6.1 事前資料登録

登録対象候補:

- 職務経歴書
- スキルシート
- 自己PR
- 志望動機
- ポートフォリオ情報
- 応募企業情報
- 求人票
- 想定質問と回答メモ

確認したいこと:

- PDFやWordをそのまま扱うか、テキスト化を前提にするか
- 資料種別ごとにどこまで構造化するか
- 要約だけで十分か、検索インデックス化まで必要か

### 6.2 リアルタイム文字起こし

対象音声候補:

- マイク入力
- システム音声
- Web会議ツールの音声

確認したいこと:

- Windowsデスクトップでシステム音声取得が安定するか
- 話者分離が必要か
- 逐次文字起こしの遅延が何秒まで許容されるか

### 6.3 質問意図解析

分類候補:

- 経歴確認
- 技術経験の深掘り
- 転職理由
- 志望動機
- 強み・弱み
- チーム開発経験
- トラブル対応経験
- キャリアプラン
- 逆質問

確認したいこと:

- 分類だけで十分か
- 意図分類と回答生成を分けた方が精度が安定するか
- どの粒度までカテゴリを増やすべきか

### 6.4 回答支援

出力候補:

- 回答の要点
- 回答例
- 話すべき実績
- 避けた方がよい表現
- 30秒版 / 60秒版の回答例

確認したいこと:

- 面接中は長文より箇条書き中心の方が見やすいか
- 回答例は全文生成より、論点サジェストの方が安全か
- 誤誘導を避けるUIが必要か

### 6.5 面接後の振り返り

レポート候補:

- 質問一覧
- 回答案の履歴
- うまく答えられなかった質問
- 次回改善ポイント
- 追加で準備すべき回答
- 職務経歴書に追記した方がよい内容

確認したいこと:

- 面接後レポートだけでも単独で価値があるか
- 面接中支援なしでもプロダクトとして成立するか

## 7. 調査観点

### 7.1 実用性

- 面接中に本当に見られるUIサイズ・情報量か
- 5〜10秒以内に回答方針を出せるか
- 実際の面接で使うと認知負荷が高すぎないか

### 7.2 倫理・運用面

- 面接中利用が企業ルールや面接の公平性と衝突しないか
- どこまでを「補助」と定義するべきか
- 隠し利用を助長する設計にならないか

### 7.3 セキュリティ

- 面接内容や職務経歴書は個人情報を含む
- ローカル保存を基本にすべきか
- 外部AI API利用時は明示同意が必要
- 音声・文字起こしデータの保存有無を選べるべき
- APIキーの安全な保存方法が必要

### 7.4 技術実現性

- ローカル完結でどこまで成立するか
- Windowsデスクトップで音声取得と低遅延処理が可能か
- RAGを入れるべきか、単純検索で十分か

## 8. 技術候補

### 8.1 アプリ基盤

- Tauri
- Electron

メモ:

- Windowsデスクトップ前提ならTauriの軽さは魅力
- 音声やネイティブ制御の実装難易度は別途比較が必要

### 8.2 音声認識

- Whisper
- faster-whisper
- OpenAI Speech to Text
- Google Speech-to-Text

メモ:

- ローカル完結ならWhisper系が有力
- 速度優先なら外部APIも比較対象

### 8.3 AI回答生成

- OpenAI
- Claude
- Ollama
- LM Studio

メモ:

- 面接中用途では速度と要約品質が重要
- 個人情報観点ではローカルLLMも有力候補

### 8.4 データ検索

- ローカルRAG
- ベクトルDB
- SQLite
- Chroma
- Qdrant

メモ:

- MVPでは単純検索や埋め込みなし構成でも始められそう
- 資料量が少ないなら過剰設計を避けたい

## 9. MVP仮説

初期版では以下に絞る案が妥当そう。

- 職務経歴書の登録
- 求人票の登録
- 面接中の文字起こし
- 質問意図の判定
- 回答案の提示
- 面接後の質問一覧出力

MVPとして追加検討:

- 面接中支援ではなく、まずは面接後の振り返り機能から入る方が安全かもしれない

## 10. 将来的な拡張候補

- 企業ごとの回答最適化
- 面接練習モード
- 想定質問の自動生成
- 回答音声の採点
- 話し方の分析
- 表情・視線分析
- 面接後レポートのPDF出力
- カレンダー連携
- 応募管理機能
- エージェント別の求人管理

## 11. 未解決論点

- 面接中利用をどこまで許容する前提にするか
- 音声取得をマイク中心にするか、システム音声取得まで含めるか
- ローカル完結を優先するか、外部API併用を前提にするか
- 回答表示を要点中心にするか、短文スクリプト中心にするか
- 事前資料の検索を単純検索で始めるか、RAGまで最初から入れるか
- 面接中支援と面接後振り返りのどちらを主価値に置くか

## 12. 次に調べること

- 類似アプリがどこまで面接中支援をやっているか
- Windowsでの音声取得手段
- Whisper系の速度と精度
- ローカルLLMで短時間応答が成立するか
- 面接中利用に関する利用規約・倫理面の整理

## 13. 類似アプリ調査

このセクションは比較表ではなく、アプリごとの調査メモを残す場所にする。

### Cluely

https://cluely.com/

調査メモ:

- 公式のクローズドソース製品。fork元としては使えない
- 公式サイトとドキュメント上では、リアルタイム音声取得、AI回答、会議ノート、ファイルアップロード、カスタム指示、不可視表示を前面に出している
- 2026-07-02時点で、公式ダウンロードページでは macOS / Windows を案内
- 価格ページでは Free / Pro があり、商用SaaSとして継続運営されている
- 実装流用ではなく、UXと機能要件の調査対象

主な見どころ:

- ベンチマーク対象として有力
- 「面接中に何をどの粒度で見せるか」のUI参考になる

気になる点:

- 公式の文言が undetectable / cheat 寄り
- クローズドソースのため、中身の実装や技術選定は追えない

ライセンス・利用形態をかみ砕くと:

- これは「サービスを使う」タイプで、「コードを触る」タイプではない
- アプリを自分向けに改造したり fork したりは基本できない
- できることは、提供された機能を利用規約の範囲で使うこと
- つまり、参考にするなら機能やUIの調査対象であって、開発ベースにはならない

### Glass

https://github.com/pickle-com/glass

調査メモ:

- GPL-3.0
- GitHub APIベースでは 2026-07-02 時点で 7.5k stars、1.1k forks、open issues 108
- 最新リリースは 2025-07-13 の `0.2.4`
- 最終 push は 2025-10-26
- README上では Electron ベース。package.json と構成を見ると、Electron + JavaScript に加えて `pickleglass_web` 配下に Next.js 系のWeb部分、SQLite、Firebase、Ollama/Whisper対応がある
- READMEに「CheatingDaddy の fork」と明記あり
- Windows は README 上で beta 扱い

主な特徴:

- 類似カテゴリの中では知名度が高く、機能も多い
- 音声、画面、AI回答、履歴周りまで一通り揃っている
- Windows対応に触れており、ローカルLLM/STT対応もある

気になる点:

- 構成がかなり重い。Electron本体に加えてWeb部分、Firebase、Express系、SQLite、音声処理が混在していて、読み解きコストが高い
- interview support に必要な「事前資料ベースでの回答支援」より、meeting/copilot 色が強い
- 最新release/pushの日時だけ見ると、直近の活発さはやや弱め

メモ:

- 実装ベースというより、機能棚卸しやWindows音声取得の参考対象

補足メモ:

- GPL-3.0 なので、配布を伴う fork は GPL 条件を踏む前提で考える必要がある

ライセンス・利用形態をかみ砕くと:

- コードを読む、動かす、改造すること自体はかなり自由
- ただし、改造版を誰かに配るなら「自分が変えた部分を含むソースも渡せる状態」にする必要がある
- 自分だけで使う改造なら比較的扱いやすい
- 逆に、「中身は閉じたまま自社製品として売る」用途とは相性が悪い

### Pluely

https://github.com/iamsrikanthnani/pluely?utm_source=chatgpt.com

調査メモ:

- GPL-3.0
- GitHub APIベースでは 2026-07-02 時点で 2.2k stars、462 forks、open issues 69
- 最新リリースは 2026-01-14 の `v0.1.9`
- 最終 push は 2026-01-14
- README と package.json から、Tauri 2 + React 19 + TypeScript + Rust + SQLite 構成
- macOS / Windows / Linux の配布物を用意している
- 機能としては、透過オーバーレイ、システム音声取得、音声入力、スクリーンショット解析、ファイル添付、会話履歴、複数LLM/STTプロバイダ設定が入っている

主な特徴:

- 軽量デスクトップアプリとしての方向性が、今回の用途にかなり近い
- Windows対応を前提にしている
- 透過オーバーレイ、グローバルショートカット、音声入力、履歴保持など、面接支援アプリの土台として必要な部品が揃っている
- ElectronではなくTauriなので、Glassより土台が軽い

気になる点:

- READMEでは privacy-first / no telemetry を強く主張している一方で、実コードには `tauri-plugin-posthog` とイベント送信コードが入っていた
- 少なくとも `app_started` と `get_license` を送る実装が存在するため、READMEの表現はそのままは信じない方がよい
- UX/メッセージングが「完全にバレないAI補助」寄りで、こちらのプロダクト方針とズレる
- interview特化ではなく、汎用の stealth assistant

メモ:

- stealth色・課金色・分析コード・不要な汎用機能を落として、面接支援専用に寄せる前提で見る必要がある
- 面接前資料登録、求人票読込、質問意図分類、面接後振り返りを追加実装するベースとしては良い

補足メモ:

- private に改造して自分だけで使う分にはGPLのハードルは比較的低い
- ただし、配布するなら GPL-3.0 の条件に従う必要があるため、商用配布前提なら最初に整理が必要

ライセンス・利用形態をかみ砕くと:

- Glass と同じく、読む・改造する・自分で使う、はやりやすい
- ただし、改造版を配るならソース提供の前提が付く
- 商用利用それ自体が即禁止というより、「配布時に閉じたまま売りにくい」ライセンス
- 個人用の検証ベースには向くが、将来クローズド製品化したいなら注意が必要

- 面接支援アプリとして必要な価値より、AI agent 連携の方に寄っている

メモ:

ライセンス・利用形態をかみ砕くと:

- これも GPL-3.0 なので考え方は Pluely / Glass とほぼ同じ
- 自分用改造はしやすい
- 配布するならソース提供前提になる
- fork の出発点には使えるが、商用で閉じたい場合は相性が悪い

### Natively

https://natively.software/

調査メモ:

- GitHub: https://github.com/Natively-AI-assistant/natively-cluely-ai-assistant
- GitHub APIベースでは 2026-07-02 時点で 1.7k stars、395 forks、open issues 44
- 最終 push は 2026-07-01
- 直近リリースは `v2.7.0` が 2026-06-05
- 公式READMEでは「Personal-Use Source License」とされており、OSSではなく source-available
- 対応環境は README 上では macOS 12+、Windows 10/11
- 技術スタックは README 上では Electron + React/Vite/TypeScript + Rust + SQLite + sqlite-vec

公開されている機能:

- 面接・会議・営業・講義などに切り替える 7 つのモード
- リアルタイム文字起こし
- 画面キャプチャ解析
- システム音声とマイク音声を分けた dual-channel 処理
- ローカル Whisper STT
- Ollama を使った完全オフライン構成
- ローカルRAGメモリ
- PDF / DOCX / TXT の reference files
- 会話履歴を管理するダッシュボード

気になる点:

- README の主張がかなり強く、マーケティング色も強い
- 「limited anonymous telemetry」と書かれており、完全 no telemetry ではない
- 利用条件は personal / educational / research / non-commercial が中心で、商用転用には別ライセンスが必要
- stealth / undetectable をかなり強く押している

メモ:

- 技術的にはかなり野心的で、面接支援アプリというより総合会話コパイロット寄り
- source-available なので、fork前提で触る場合はライセンス確認が必須

ライセンス・利用形態をかみ砕くと:

- 「ソースは見せるが、自由なOSSではない」というタイプ
- 個人利用、学習、研究、非商用なら触ってよい
- ただし商用利用、SaaS化、競合サービス化、クライアント案件流用は原則だめ
- public fork も同じライセンスで出す必要があり、かなり制約は強い
- つまり、読む・試す用途には使えるが、将来の商用ベースとしては慎重に扱う必要がある

- 守るべきことは、元の著作権表示やライセンス表記を残すこと、変更したことを分かるようにすること
- 特許まわりの条項もあり、MITより少しきちんとしている
- もし本当に Apache-2.0 で確定なら、今回見た中ではかなり扱いやすい部類

## 14. ライセンス・利用形態メモ

- Glass / Pluely / Freely は GPL-3.0
- OpenCluely は LICENSE 実体は Apache-2.0 だが、README バッジは MIT、`package.json` は ISC で表記ゆれがある
- Natively は open source ではなく personal-use source license
- Interview Copilot はライセンス明記が弱い
- Cluely / LockedIn AI / Final Round AI / ParakeetAI / Interview Coder はクローズドソースの商用サービス

配布や fork を考えるときのメモ:

- GPL系は配布時の条件整理が必要
- source-available 系は非商用・個人利用条件の確認が必要
- closed-source SaaS はそのまま使う以外の自由度は低い

かみ砕いた覚え方:

- `GPL-3.0`
  自分で使うぶんには比較的触りやすい。ただし、改造版を誰かに配るならソースも開示できる前提で考える。将来クローズド製品にしたい場合は重い。

- `Apache-2.0`
  かなり使いやすい。商用利用、改造、再配布、クローズド製品への組み込みがしやすい。著作権表示やNOTICEの扱いは守る。

- `MIT`
  Apache-2.0 と同じくかなり緩い。著作権表示とライセンス文を残せば使いやすい。実務感では「かなり流用しやすい」側。

- `source-available`
  コードは見えるが、OSSではない。読んだり個人利用で試したりはできても、商用化・SaaS化・再販売は止められていることが多い。

- `closed-source SaaS`
  コードを触る前提ではない。fork不可。使うだけ。

- `ライセンス不明`
  一番扱いにくい。公開されていても、勝手に使ってよいとは限らない。参考閲覧止まりで考えるのが安全。

今回の用途に引き寄せると:

- 「自分だけで試作する」なら GPL でも source-available でもそこまで困りにくい
- 「将来配る」「商用化する」「自分の製品として閉じたい」なら Apache-2.0 / MIT の方が圧倒的に楽
- closed-source SaaS は実装ベースではなく、企画・機能調査の対象

参考:

- https://www.gnu.org/licenses/gpl-3.0.html

## 15. fork 元として見る場合のメモ

2026-07-02 時点の暫定結論:

- `Pluely` と `freely` の二択なら、fork 元としては `Pluely` の方が無難
- ただし「課金・ライセンス・解析まわりを最初から薄くしたい」なら、二択より `OpenCluely` の方が出発点として綺麗な可能性がある

理由:

- `Pluely`
  - 2026-01-14 時点まで更新されており、スター数・利用者認知・機能の厚みがある
  - Tauri + React + Rust + SQLite で構成が比較的整理されている
  - 一方で、実コード上に license activation、Pro 前提の文言、analytics が残っているため、商用化や独自化には掃除が必要

- `freely`
  - README 上は Pluely fork と明記されている
  - agent backend 方向の拡張は面白い
  - ただし 2026-03-24 以降の更新が見えず、規模も小さい
  - コード上にも PostHog、license status、Pluely 文言など fork 元の名残がまだ残っており、「Pluely の有料化を避けるための綺麗な素体」とまでは言いにくい

- `OpenCluely`
  - Apache-2.0 系で扱えるなら、配布・商用化・クローズド化の自由度が高い
  - technical interview 向けで今回のテーマに近い
  - `Pluely / freely` より機能量では劣る可能性があるが、余計な課金導線や継承負債を避けやすい

実務寄りに言うと:

- 早く形にしたいなら `Pluely` を fork して、課金・ライセンス・analytics を削る方が現実的
- 将来配布や商用化まで見据えるなら、GPL-3.0 の重さを踏まえて `OpenCluely` も強く検討した方がよい
- `freely` は fork 元というより、「Pluely 派生で何を削り、何を足したか」を読む参考先として見るのがよさそう

## 16. 類似アプリを fork 元として選ぶならどれか

2026-07-02 時点で、fork して自分の面接支援アプリを作る前提なら、候補は大きく3段階に分かれる。

第一候補:

- `OpenCluely`

第二候補:

- `Pluely`
- `Glass`

第三候補:

- `freely`
- `Interview Copilot`
- `Natively`

除外寄り:

- `Cluely`
- `LockedIn AI`
- `Final Round AI`
- `ParakeetAI`
- `Interview Coder`

理由をアプリ別に整理すると:

### OpenCluely を使う場合

向いているケース:

- 将来、配布・商用化・クローズド化も視野に入れたい
- まずは technical interview 向けのデスクトップ補助を作りたい
- 課金やライセンス認証の残骸を掃除するコストを減らしたい

良い点:

- LICENSE 実体が Apache-2.0 なので、今回見た中ではかなり扱いやすい
- 2026-07-01 まで push があり、直近リリースも 2026-06-29 と新しい
- Electron ベースで構成が比較的追いやすい
- 音声、スクリーンショット、ストリーミング回答、セッションメモリなど、核になる機能は揃っている
- README / コード上では telemetry や課金導線が見えず、素の土台として扱いやすい

弱い点:

- technical interview 特化が強く、一般面接や事前資料管理は薄い
- Gemini 依存の色が比較的強い
- README は MIT、`package.json` は ISC、LICENSE は Apache-2.0 で表記ゆれがある

総評:

- 一番「自分のアプリの土台」にしやすい
- ただし fork 直後にやるべきこととして、ライセンス表記の整理、依存プロバイダの抽象化、面接前後フローの追加、は最初から見込んだ方がよい

### Pluely を使う場合

向いているケース:

- まずは最短で MVP を作りたい
- Tauri + React + Rust + SQLite の構成で進めたい
- 多少の掃除コストを払っても、完成度の高い土台を優先したい

良い点:

- 2026-01-14 まで更新されており、機能の厚みがある
- 軽量で、デスクトップ補助アプリとしての構成が分かりやすい
- OS 配布や UI の雛形が揃っていて、着手が速い

弱い点:

- GPL-3.0 なので、将来クローズド化や商用配布を考えるなら重い
- 実コード上に license activation、Pro 導線、PostHog analytics が残っている
- そのまま fork すると、「有料化済み fork を掃除している」状態から始まる

総評:

- 最短で試作するなら強い
- ただし「自分の製品」に育てる前提では、初手から掃除対象が多い
- 商用や再配布まで見据えるなら、技術的には良くてもライセンス面で第一候補にはしにくい

### Glass を使う場合

向いているケース:

- 面接支援だけでなく meeting assistant 側まで広げたい
- ローカル保存、クラウド同期、Web 管理画面のような厚い構成も視野に入れている
- 大きめのコードベースでも読める

良い点:

- 2025-10-26 まで push があり、stars も多く認知が高い
- Electron 本体に加えて、`pickleglass_web` や backend もあり、拡張余地は広い
- OpenAI / Gemini / Anthropic / Ollama / Whisper など複数プロバイダを持つ
- 会議サマリ、履歴、パーソナライズ方向に広げやすい

弱い点:

- GPL-3.0
- Electron + Web + backend + Firebase + SQLite で構成が重く、fork 元としてはかなり大きい
- billing、auth、cloud sync 方向の名残もあり、面接支援だけを切り出すには過剰

総評:

- 「最も機能を盛りやすいベース」ではある
- ただし小さく始めるには重い
- interview helper というより、meeting intelligence 系まで作りたいときの候補

### freely を使う場合

総評:

- `Pluely` から有料化部分を避けたくて見る先としては理解できる
- ただし実態は `Pluely` の名残がまだ多く、更新も弱い
- fork 元として第一候補にはしにくい

### Interview Copilot を使う場合

総評:

- 発想は分かりやすく、serverless / browser-only の参考にはなる
- ただし Vue2 の古めな pure web app で、desktop stealth 系の基盤にならない
- ライセンスも明確でないため、fork 元としては弱い

### Natively を使う場合

総評:

- 実装の厚みはかなりある
- ただし personal-use source license なので、商用化・SaaS化・競合製品化・公開 fork に強い制限がある
- 読む参考にはなるが、自分のサービスの土台としては基本的に不向き

結論を短く言うと:

- 将来の自由度まで含めて一番無難なのは `OpenCluely`
- 最速で MVP を出したいなら `Pluely`
- 面接支援を超えて会議支援や履歴管理まで厚くやりたいなら `Glass`

この調査メモの目的に最も近い選び方:

- 「面接中の補助」から入り、あとで「事前資料登録」「面接後振り返り」まで広げたいなら、まず `OpenCluely` を土台にして足すのが最もきれい
- 「まずはすぐ動くものを作り、あとで捨ててもよい」なら `Pluely` を掃除して試作するのが速い
- `freely` は fork 元というより、`Pluely` 派生の調査対象として扱うのがよい

### この前提が「自分だけ使う・販売しない・配布しない」場合

この条件なら、選定は少し変わる。

第一候補:

- `Pluely`

第二候補:

- `OpenCluely`

第三候補:

- `Glass`

理由:

- `Pluely`
  - 自分だけ使うなら GPL-3.0 の重さはかなり気にしなくてよい
  - 既に機能が揃っており、最短で「使える自分用ツール」にしやすい
  - 課金・ライセンス・analytics の残骸はあるが、自分用ならまずは不要部分を削る方向で進めやすい

- `OpenCluely`
  - こちらの方が土台はきれい
  - ただし初期機能は `Pluely` より薄めなので、自分用として早く実用に持っていくなら一歩劣る

- `Glass`
  - 高機能だが重い
  - interview helper を作るには構成が少し大きすぎる

この前提での結論:

- 「まず自分用として早く完成させたい」なら `Pluely`
- 「少し遠回りでも、綺麗な構成から自分用に育てたい」なら `OpenCluely`
- `Natively` は自分用ローカル改造の余地はあるが、license 条件が強く、license checks や commercial まわりの制限もあるので、土台としては積極的には勧めにくい

## 17. fork する上での注意点

前提:

- ここでは「自分だけで使う」「販売しない」「配布しない」を前提に整理する
- そのため、配布時の厳しい条件よりも、実務上ハマりやすい点を優先して見る

まず大事なこと:

- `fork` しただけでは、そのまま安心して使える状態にはならない
- とくにこの系統のアプリは、課金、analytics、API key、stealth、OS 権限、外部サービス依存が絡みやすい

注意点は次の通り。

### 1. ライセンスは「配らないなら軽くなる」が、「何でもしてよい」ではない

- `Pluely` と `Glass` は GPL-3.0
  - 自分だけで改造して使うぶんには、かなり扱いやすい
  - ただし将来もし誰かに配る気が出た時点で、ソース開示や同ライセンス継承の整理が要る

- `OpenCluely` は LICENSE 実体は Apache-2.0
  - 自分用でも扱いやすいし、将来方針が変わっても比較的柔らかい
  - ただし README や `package.json` の表記ゆれは、fork 後に自分側で整理した方がよい

- `Natively` は personal-use source license
  - 自分用で読む・改造する余地はある
  - ただし license checks や attribution、再利用制限が強く、気軽な土台にはしにくい

### 2. fork 直後に「不要機能の掃除」を先にやる

- `Pluely`
  - license activation
  - Pro 文言
  - analytics / PostHog
  - 支払い導線や premium 条件分岐

- `freely`
  - `Pluely` 由来の文言や license status の残骸
  - analytics まわり

- `Glass`
  - auth / billing / Firebase / Web 管理画面など、面接支援に不要な層

これを最初にやらないと:

- どこが本体機能で、どこが商用化の残骸か分かりにくくなる
- バグを追うときに余計な分岐が増える
- 「自分用にしたいだけ」なのに修正コストが膨らむ

### 3. API key と telemetry の扱いを最優先で確認する

- `.env`
- OS keychain
- local DB
- analytics SDK
- remote logging

このあたりは fork 後すぐ確認した方がよい。

特に確認する点:

- API key がどこに保存されるか
- 音声やスクリーンショットが外部送信される経路があるか
- analytics が初期状態で有効か
- エラー送信や usage 計測が残っていないか

自分用でも、ここを見落とすと「ローカルで使っているつもりが一部は外へ出ていた」という形になりやすい。

### 4. 依存プロバイダ固定を早めに緩める

- `OpenCluely` は Gemini 色が強い
- `Pluely` は provider 切替はあるが、商用化由来の分岐が混ざっている
- `Glass` は複数 provider があるが、構成が大きい

将来自分の使い方に合わせるなら:

- LLM provider
- STT provider
- screenshot 処理
- prompt 管理

この4つは早めに分離した方がよい。

### 5. OS 権限と stealth 動作は fork 後に必ず自分で検証する

この系統のアプリは:

- 画面キャプチャ権限
- マイク権限
- グローバルショートカット
- overlay 表示
- screen share 非表示挙動

が OS やバージョン差分の影響を受けやすい。

つまり:

- upstream で動く
- README に書いてある

だけでは足りず、自分の環境で確認が必要。

### 6. upstream 追従しやすい形で fork する

- 最初から大規模リネームしすぎない
- まずは不要物削除と設定切り出しを優先する
- 自分の変更点を `PATCHES.md` などに残す

こうしておくと:

- upstream の修正を取り込みやすい
- 何を消したか後から分かる
- fork がすぐ別物になって破綻するのを防ぎやすい

### 7. 自分用でも「元の名前・ブランド」は早めに外す

- アプリ名
- アイコン
- README
- package name
- process title

これは法務だけでなく、混乱防止の意味でも早めに変えた方がよい。

### 8. 結局いちばん実務的な注意点

- `Pluely` を fork するなら「不要な商用化残骸を最初に消す」
- `OpenCluely` を fork するなら「ライセンス表記ゆれと provider 依存を最初に整理する」
- `Glass` を fork するなら「機能が多すぎるので、まず面接支援に要らない層を削る」
- `Natively` は読んで参考にするのはよいが、fork 元としては慎重に扱う

## 18. Pluely を無料で触るための local STT provider メモ

前提:

- `Pluely` は custom STT provider を `curl` ベースで追加できる
- `{{AUDIO}}` を含む `curl` を設定し、レスポンスから文字列を抜く path を指定する仕組み
- そのため、`OpenAI 互換の /v1/audio/transcriptions` を出す local STT server が最も相性がよい

実務上の第一候補:

- `Speaches`

理由:

- OpenAI API compatible の STT/TTS server
- `/v1/audio/transcriptions` をそのまま持っている
- CPU 版 Docker で起動できる
- `Pluely` の custom STT provider に最もそのまま繋ぎやすい
- STT は faster-whisper ベース

公開情報メモ:

- docs では Docker Compose が推奨
- 単体 Docker でも `ghcr.io/speaches-ai/speaches:latest-cpu` を起動可能
- speech-to-text docs では `http://localhost:8000/v1/audio/transcriptions` を案内
- モデルは初回利用時または明示ダウンロード時に取得

Pluely 側の設定イメージ:

```bash
curl -X POST "http://localhost:8000/v1/audio/transcriptions" \
  -H "Authorization: Bearer dummy" \
  -F "file={{AUDIO}}" \
  -F "model=Systran/faster-distil-whisper-small.en"
```

- `Response Content Path` は `text`

良い点:

- `Pluely` との接続が素直
- OpenAI Whisper 系の mental model で理解しやすい
- API server として切り出せるので、将来自作アプリでも流用しやすい

弱い点:

- GUI アプリというより server
- Docker 前提で考えるのが楽
- 初回モデル取得はそれなりに時間と容量を使う

第二候補:

- `GPUStack + vox-box`

理由:

- `vox-box` は OpenAI audio API compatible
- Windows / macOS は desktop installer 推奨で、GUI 寄りに触りやすい
- docs 上でも Windows / macOS / Linux 対応が見える

良い点:

- `LM Studio` 的な「入れて管理する」感覚に近い
- `vox-box` 自体は STT/TTS 両方を扱える
- モデル管理まで含めて厚め

弱い点:

- `Pluely` の STT 用途だけなら少し大げさ
- 単純な local STT server が欲しいだけなら `Speaches` の方が軽い

第三候補:

- `whisper.cpp server`

理由:

- 軽量でローカル STT の定番
- web server もある

弱い点:

- 導入は `Speaches` より面倒
- ビルドやモデル準備の負担がある
- 「とりあえずすぐ使う」には向きにくい

今回の用途では外れるもの:

- `Handy`

理由:

- `Handy` は local STT desktop app としては優秀
- ただし `Pluely` に繋ぐための `STT provider` / `localhost API server` ではない
- 単体で dictation する用途には向くが、`Pluely` の provider として使う用途には噛み合いにくい

この前提での結論:

- `Pluely` を無料で触るために local STT を楽に繋ぎたいなら、まずは `Speaches`
- `GUI で管理したい` 気持ちが強いなら `GPUStack + vox-box`
- `Handy` は便利だが、今回ほしい部品とは少し違う
